### 致命错误6：流式输出没开启导致界面卡死

**错误表现：** 调用o3-mini后，整个程序卡在那里，很久才响应，或者直接超时报错。

**错误原因：** 默认情况下，很多SDK（比如OpenAI Python库）是“非流式”模式，它会等待o3-mini全部生成了完整回答后才一次性返回。这会导致长达几十秒的等待时间，尤其网络不好的时候容易断连。

**完美解法：** 在调用时显式开启`stream=True`：
python
response = client.chat.completions.create(
    model="o3-mini",
    messages=[...],
    stream=True  # 开启流式输出
)
for chunk in response:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")

这样就能逐字符输出，用户体验瞬间提升。

### 致命错误7：Token限制没设对导致意外扣费

**错误表现：** 调用了o3-mini后，发现消耗Token量巨大，或者回答被截断，或者报错“max_tokens is too large”。

**错误原因：** 很多新手不知道设置`max_tokens`，默认值很大或很小。如果是大模型自动生成大量内容，你的余额可能在一次对话中就被掏空。

**完美解法：** 根据自己的需求，明确设置`max_tokens`（最大输出Token数）。比如：
python
response = client.chat.completions.create(
    model="o3-mini",
    messages=[...],
    max_tokens=1024  # 控制输出长度
)

如果你只是问个简单问题，设128就够了；深度的技术回答可以设2048。同时，建议在千聚后台设置每日消费限额，防止超支。

### 致命错误8：用错`messages`格式

**错误表现：** 传入的`messages`结构不对，比如用`system`角色替代了`user`，或者`content`写成字符串而非列表。

**错误原因：** o3-mini对输入格式要求非常严格。正确的格式应该是列表，每个元素是一个字典，包含`role`和`content`。

**完美解法：** 严格遵循以下结构：
python
messages = [
    {"role": "system", "content": "你是一个专业的编程助手。"},
    {"role": "user", "content": "请用Python写一个快速排序函数。"}
]

`role`有三个值：`system`（系统提示）、`user`（用户输入）、`assistant`（模型回答）。`content`必须是字符串，不能是列表或数字。

**错误原因：** 开发者没有为API调用添加任何容错机制。任何线上服务都可能有瞬间的高负载或网络抖动。

**完美解法：** 添加重试逻辑和错误码判断。使用`tenacity`库：

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
def call_o3_mini(messages):

        model="o3-mini",
        messages=messages,
        max_tokens=1024
    )
    return response.choices[0].message.content
