### 致命错误二：直接使用“国内直连”就以为万事大吉，忽略网络稳定性

看到[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)（www.qianjuai.com）支持国内直连，部分开发者觉得接入后就能稳定运行了，于是直接在生产环境里用最简单的 HTTP 客户端调用，不做任何重试或超时控制。

问题在于，虽然直连避免了“翻墙”的麻烦，但“直连”不等于“绝对不掉线”。网络波动、服务器瞬时负载、DNS 解析延迟，这些都会导致偶发的请求失败。有开发者在测试环境用得很开心，上线第一天用户一多，频繁出现 `ConnectionError` 和超时，整个服务的可用性直接拉胯。

| 做法 | 后果 |
| :--- | :--- |
| 无重试机制 | 一次失败即服务崩溃 |
| 无超时设置 | 请求卡死，消耗线程资源 |
| 最佳做法：重试 + 指数退避 + 超时控制 | 服务可用性大幅提升 |

**如何避免：**
不管你在国内直连多么顺畅，调用 API 时都必须做好容错。推荐使用 `tenacity` 或者简单的 Python `retry` 装饰器，设置 `max_retries=3`，并且使用指数退避（比如重试间隔 1s、2s、4s）。同时，一定要在请求里设置 `timeout`，比如 `openai.timeout=30`，避免请求无限等待。[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)的 API (https://www.qianjuai.com/v1) 本身很稳定，但你自己的客户端逻辑要扛得住偶尔的网络抖动。

### 致命错误四：调用 DeepSeek-V3 时，把 `response_format` 参数写错

这个错误在官方文档里明确写了，但很多新手还是掉进坑里。

DeepSeek-V3 的 API 文档里明确指出，如果你要用 JSON 模式（`response_format={"type": "json_object"}`），有两个前提：第一，你必须传入 `json_object` 作为 type；第二，**系统消息（system message）里必须明确告诉模型返回什么 JSON 格式**。很多新手只记得第一条，忽略了第二条。

比如，你写了 `response_format={"type": "json_object"}`，但 System Message 只写了一句 “你是一个有用的助手”，没有说明要输出 JSON 对象。此时，模型可能依然返回一段普通文本，而不是预期的 JSON，导致你的代码解析失败，抛出 `JSONDecodeError`。

| System Message 内容 | 结果 |
| :--- | :--- |
| “你是一个有用的助手” | API 可能返回普通文本 |
| “请以 JSON 格式返回结果，包含 `name` 和 `age` 字段” | 模型正确返回 JSON |

**如何避免：**
在调用[[千聚ai官网](https://www.qianjuai.com/)](https://www.qianjuai.com/)的 DeepSeek-V3 接口时，如果你需要 JSON 输出，必须先在 System Message 里明确写出具体的 JSON 结构和要求。举个例子：
`{"role": "system", "content": "你是一个信息提取助手。请根据用户输入的句子，以 JSON 格式返回，包含字段：name（姓名）、age（年龄）。"}`
然后在 `response_format` 里设置 `{"type": "json_object"}`。这样模型才会乖乖听话。如果希望在任何情况下都拿到稳定格式，还是推荐使用 `Function Calling` 或者 `response_format` 配合 `json_schema`。
