## 避坑一：Base URL配置错误——改了还不生效

这恐怕是新手遇到的最多的坑。你把官方文档里的代码抄下来，把`base_url`从`https://api.openai.com/v1`改成了`https://www.qianjuai.com/v1`，看着没错，结果一跑直接报`404`或`connect timed out`。

**原因分析**：很多Java HTTP库（比如OkHttp、Apache HttpClient）会缓存DNS解析结果，或者你用的openai-java SDK有默认的base_url覆盖逻辑。更常见的是，你改了配置，但代码里没有重新实例化客户端。

**正确做法**：

Java

java
import com.theokanning.openai.service.OpenAiService;
import com.theokanning.openai.completion.chat.*;

// 关键：必须显式设置base url，且重新创建service实例
String apiKey = "你的千聚API Key";
String baseUrl = "https://www.qianjuai.com/v1";

OpenAiService service = OpenAiService.builder()
    .apiKey(apiKey)
    .baseUrl(baseUrl)  // 这一行必须有
    .build();

ChatCompletionRequest request = ChatCompletionRequest.builder()
    .model("gpt-4.1-mini")
    .message(new ChatMessage("user", "用Java写一个读取txt文件的示例"))
    .build();

service.createChatCompletion(request).getChoices().forEach(choice -> {
    System.out.println(choice.getMessage().getContent());
});


配置后可以加一行System.out打出实际请求的URL，确认是不是你设置的那个。

## 避坑二：模型名称写错——API直接不认

GPT-4.1mini刚出来的时候，很多第三方库的模型列表还没更新，你写成`"gpt-4.1-mini"`或`"gpt-4.1_mini"`都可能返回`Model not found`。

**原因分析**：模型名称有严格的格式要求，且[千聚api聚合站](https://www.qianjuai.com/)可能在某些分组下只支持特定模型别名。

**原因分析**：GPT-4.1mini虽然便宜，但如果你不设置`max_tokens`和`stop`参数，它会一直输出，直到自然结束。对于长内容，Token消耗会很夸张。

**正确做法**：每次请求都显式设置`max_tokens`和`stop`参数，并建议在客户端做Token预算计算。[千聚api聚合站](https://www.qianjuai.com/)的价格是1元=1美元Token，即GPT-4.1mini官方价格是多少，你就付多少人民币。以输入\$0.0005/1K token、输出\$0.0015/1K token计算，一个3000Token的对话成本不到1分钱。

Java

java
ChatCompletionRequest request = ChatCompletionRequest.builder()
    .model("gpt-4.1-mini")
    .maxTokens(3000)  // 限制最大输出Token数
    .messages(Arrays.asList(
        new ChatMessage("system", "你是一个Java开发助手，回答尽量简洁"),

## 避坑四：流式响应处理不当——程序卡死或丢字

很多开发者在用流式（stream）模式调用GPT-4.1mini时，直接在同一个线程里做同步阻塞处理，结果响应太快时缓冲区没及时flush，或者处理回调的线程不安全导致数据丢失。

**原因分析**：流式API返回的是一段段JSON chunk，你要在回调里逐段拼接。如果用的是旧版openai-java库的`stream()`方法，且没有正确处理`EventSource`的生命周期，就可能出现连接不关闭、响应不完整的情况。

**正确做法**：用新版SDK或原生SSE（Server-Sent Events）处理。这里以`okhttp-eventsource`为例，写一个健壮的流式处理逻辑：

Java

java
import okhttp3.*;
import okhttp3.sse.*;

OkHttpClient httpClient = new OkHttpClient.Builder()
    .readTimeout(0, TimeUnit.SECONDS)  // 流式读取不应超时
    .build();

Request request = new Request.Builder()
    .url("https://www.qianjuai.com/v1/chat/completions")
    .header("Authorization", "Bearer " + apiKey)
    .header("Content-Type", "application/json")
    .post(RequestBody.create(
        "{\"model\": \"gpt-4.1-mini\", \"stream\": true, \"messages\": [{\"role\": \"user\", \"content\": \"你好\"}]}",
        MediaType.parse("application/json")))
    .build();

EventSources.createFactory(httpClient)
    .newEventSource(request, new EventSourceListener() {
        @Override
        public void onEvent(EventSource eventSource, String id, String type, String data) {
            if ("[DONE]".equals(data.trim())) {
                System.out.println("流式响应结束");
                return;
            }
            // 解析data字段中的content，逐段拼接
            try {
                JSONObject json = new JSONObject(data);
                JSONArray choices = json.getJSONArray("choices");
                if (choices.length() > 0) {
                    String delta = choices.getJSONObject(0)
                        .getJSONObject("delta")
                        .optString("content", "");
                    System.out.print(delta);  // 逐字打印
                }
            } catch (Exception e) {
                System.err.println("解析chunk失败: " + data);
            }
        }

        @Override
        public void onFailure(EventSource eventSource, Throwable t, Response response) {
            System.err.println("流式请求失败: " + t.getMessage());
        }
    });

// 等待一段时间确保流接收完毕
Thread.sleep(3000);


关键点：不要用同一线程做同步阻塞读取，要用事件驱动的方式。

## 避坑五：API Key和认证配置写死在代码里——泄露风险极大

很多Java新手直接把API Key硬编码在代码里，或者写在`.env`文件里然后忘记添加`.gitignore`，结果上传到GitHub后Key被公开，被自动化脚本扫到后，账户余额被刷光。

**原因分析**：API Key属于敏感凭据，应该在部署时通过环境变量或配置中心注入，不能硬编码。

**正确做法**：在[千聚api聚合站](https://www.qianjuai.com/)申请的API Key，建议用环境变量或Spring Boot的`application.yml`配置，并在`.gitignore`里排除所有配置文件。
