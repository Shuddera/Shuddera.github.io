## 为什么不用官方 SDK，要自己手写？

官方 SDK 的初衷是好的——封装了鉴权、序列化、异常处理，拿来即用。但在实际生产中，我们遇到的痛点非常真实：

1.  **依赖臃肿**：为了一个简单的 AI 调用，SDK 往往引入几十个间接依赖，与 Spring Boot 版本冲突是常有的事。
2.  **无法精细控制**：SDK 的底层连接池和超时策略是写死的，碰到网络抖动，一个默认的 10 秒超时能让你的整个链路堵死。
3.  **调试成本**：虽然调用模型本身是付费的，但“零成本”指的不是 Token 免费，而是**你不需要额外花钱买网关、不需要升级服务器配置**。

手写 HttpClient，本质上是在做减法：去掉所有你不需要的封装，只保留请求-响应的核心逻辑。这就是快的原因。

第二步：手写一个“极简” HttpClient

这里我们直接上干货。在 Java 中，最原生的方式就是使用 `java.net.http.HttpClient`（JDK 11+ 内置），不需要引入任何第三方库。

**关键代码切片（纯注释已为你写好）：**

java
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.net.http.HttpResponse;
import java.time.Duration;

public class SimpleErnieClient {

    // 注意：这里使用的是[千聚ai大模型聚合站](https://www.qianjuai.com/)的中转地址，完全兼容文心API
    private static final String BASE_URL = "https://www.qianjuai.com/v1/chat/completions";

    // 你的API Key
    private static final String API_KEY = "sk-你的Key";

    public static void main(String[] args) throws Exception {
        HttpClient client = HttpClient.newBuilder()
                .connectTimeout(Duration.ofSeconds(5))  // 连接超时短一点，避免死等
                .build();

        // 构建请求体（这里使用文心的ERNIE-3.5模型）
        String jsonBody = "{"
                + "{\"model\":\"ernie-3.5\",\"messages\":[{\"role\":\"user\",\"content\":\"使用Java手写一个快速排序\"}]}";

        HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create(BASE_URL))
                .header("Content-Type", "application/json")
                .header("Authorization", "Bearer " + API_KEY)
                .timeout(Duration.ofSeconds(30))  // 读超时
                .POST(HttpRequest.BodyPublishers.ofString(jsonBody(jsonBody))
                .build();

        HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());

        System.out.println("状态码: " + response.statusCode());
        System.out.println("响应体: " + response.body());
    response.body());
    }
}

**为什么会快在这几个环节？**

1.  **消除 Serialization 开销**：SDK 为了支持各种模型和参数，内部做了一次复杂的 JSON 序列化封装，而手写代码直接传字符串，省去了这一步。
2.  **连接复用**：官方的自定义连接池在某些场景下会创建新连接，而 JDK 原生的 `HttpClient` 只要保持长连接，复用率极高。
3.  **异常处理短路**：SDK 捕获到错误后，往往还会做一些重试或日志记录，手写版本直接返回，交给应用层处理。
