## 陷阱二：忽视参数设置，单次调用多花2-3倍Token

官方Python示例里最常见的坑就是“裸参数”。你看看一份典型的通义千问调用代码：

python
response = client.chat.completions.create(
    model="qwen-max",
    messages=[
        {"role": "system", "content": "你是一个帮助用户解答问题的助手。"},
        {"role": "user", "content": "请用500字以内解释什么是深度学习。"}

    # 注意：没有设置max_tokens，temperature等参数
)


这段代码跑起来，默认行为是：**输出尽可能多的token**（一般是4K上限），temperature==1.0（高随机性），frequency_penalty==0（不重复惩罚）。结果就是你希望500字，它给你来1500字，多出的1000字白白计费。如果对话中包含长system prompt，每次调用还会重复发送，token消耗直接翻倍。

真正的费用杀手是“无效重复”：每次请求都带着相同的上下文，但模型无法复用缓存，相当于同一段文字计费N次。

## 陷阱三：忽略“重试机制”，失败一次白烧全部费用

通义千问API在高峰期（尤其是国内上班时间9-11点、下午2-4点）经常返回5xx错误或超时。很多开发者写的代码如此：

python
try:
    response = client.chat.completions.create(
        model="qwen-max",
        messages=[{"role": "user", "content": "你好"}]
    )
    process_result(response)
except Exception as e:
    print("调用失败：", e)
    # 然后什么也不做


犯错了。因为 `try-except` 直接捕获异常，如果是网络超时或服务端错误，这次请求的Token已经在路上消耗了（很多情况下服务端已开始处理），但你一个字都没收到。等于白白烧钱。

更可怕的情况是：你写了循环重试：

python

    try:
        response = client.chat.completions.create(...)
        break
    except:
        continue
